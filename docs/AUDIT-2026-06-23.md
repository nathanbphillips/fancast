## 4. Findings — prioritized

### HIGH (verified, fix recommended before wiring post-login deep-links)

**H-A. ✅ FIXED — Open-redirect regression (was AUDIT.md M-2) — `safeNextPath` bypassed by control-char-injected authority**
> Fixed in `lib/redirect.ts`: reject any char ≤0x20/0x7f, require single-leading-slash input, re-parse same-origin, AND re-check the normalized **output** starts with a single `/` (closes the `/..//` traversal bypass). `scripts/redirect-test.ts` extended to 20 cases incl. `%09`/`%0A`/`%0D`/space/`/..//`. Adversarially fuzzed (271k+ inputs, unicode confusables, authority-smuggling) — 0 origin escapes.

`lib/redirect.ts:7-11`, `app/auth/callback/route.ts:19,61`, `scripts/redirect-test.ts:25`
`safeNextPath` only checks `raw[0] !== '/'` and `raw[1] === '/'|'\\'`. The WHATWG URL parser strips leading C0/whitespace control chars before parsing, so `?next=/%09//evil.com` (decodes to `/\t//evil.com`) passes the guard and `new URL(next, origin)` resolves to `https://evil.com/`. Confirmed end-to-end (reproduced in Node) for `%09`, `%0A`, `%0D`, and even the single-slash `/\t/evil.com`. Worse: `scripts/redirect-test.ts:25` asserts the vulnerable output as a PASS — the test certifies the hole.
- **Why HIGH:** post-login phishing/credential-redirect vector.
- **Currently hard to reach:** `next` is never threaded from the sign-in form into the OAuth/magic-link redirect (see finding L-? below), so today it's only reachable via an externally crafted callback URL. It becomes live the moment deep-link-after-login is wired (Phase 9 Resend emails).
- **Fix:** after building `new URL(raw, origin)`, require `url.origin === origin` and return only `url.pathname + url.search + url.hash`; OR strip `[\x00-\x20]` and re-validate that the result starts with a single `/` not followed by `/`/`\`. Correct `scripts/redirect-test.ts:25` and add cases for `/%09//`, `/%0A//`, `/%0D//`.

### MEDIUM (hardening; bounded blast radius in single-commentator MVP)

**M-A. ✅ FIXED — SSRF residual: DNS-rebinding TOCTOU in the unfurler** — `lib/unfurl.ts`
> Fixed: `resolvePinned` validates + returns the address, and `rawRequest` pins it into the connection via the node `lookup` option (SNI preserved via `servername`), so there is no second independent resolution; every redirect hop re-validates + re-pins. The fix also closed a **pre-existing IPv6 gap** the review surfaced: `isPrivateIp` now expands IPv6 to 8 groups (`v6Groups`) and decodes embedded-IPv4 for `::ffff:0:0/96`, `::/96`, and `64:ff9b::/96` (NAT64) — hex-mapped/compat/NAT64 literals are now blocked. Verified by `test:ssrf` + a 42-case matrix. Original detail:

`assertPublicUrl` resolves+validates DNS, then `fetch` resolves again independently; the validated IP isn't pinned. A short-TTL host can return a public IP at validation and a private IP (e.g. 169.254.169.254) at fetch. Literal/encoded/redirect vectors are closed (H-2 fix verified, 24/24 ssrf tests pass) — rebinding is the remaining gap.
- **Fix:** resolve once, verify public, then connect to the pinned IP (custom `lookup` on an undici dispatcher that re-checks the resolved address at connect time).

**M-B. ✅ FIXED — Public stats proxy has no rate limit or fixture allowlist** — `app/api/stats/[fixtureId]/route.ts`
> Fixed: the route now looks the id up in the synced `fixtures` table and returns `emptyStats(id)` (no upstream call) for any id not present. Rooms can only reference synced fixtures (FK), so legit polling is unaffected; id-enumeration can no longer bypass the per-id cache to amplify Sportmonks calls. (Re-audit confirmed fails-closed on cache-poisoning, RLS false-negative, and error paths.)

GET is public and accepts any positive integer id; the 10s cache is per-id, so enumerating ids bypasses the cache and triggers one fresh Sportmonks fetch each — unauthenticated amplification against the metered plan.
- **Fix:** constrain to known fixtures (look the id up in the synced `fixtures` table; return `emptyStats(id)` otherwise) and/or a light per-IP rate limit. Keeps reading open.

**M-C. ◐ MITIGATED — recordings privacy rests only on the bucket `public:false` flag** — `lib/egress.ts`, `app/api/recordings/route.ts`
> Mitigated, not fully closed (honest status). `ensureRecordingsPrivate` re-asserts `public:false` whenever the downloads panel is read (on top of every Start Broadcast), shrinking the public-toggle exposure window. **Note:** the audit's suggested `storage.objects` RLS policy was deliberately NOT added — a *public* Supabase bucket serves `/object/public/...` with **no RLS**, so RLS cannot guard this threat; re-asserting private is the actual control. Residual: the re-assert is best-effort and only fires on authorized reads, so a manual public-toggle with no intervening read/broadcast persists. **Phase 10 full closure:** store recordings under an unguessable path token so the public path isn't guessable even if the bucket is toggled. Original detail:

Table RLS exists, but the audio bytes are protected only by the private-bucket flag (re-asserted only on the next Start Broadcast). A manual dashboard toggle / bucket-name reuse between broadcasts would expose `/storage/v1/object/public/recordings/{roomId}/full.mp3`. Touches golden rule 1 (recordings privacy).
- **Fix:** add a `storage.objects` RLS policy restricting `select` on `bucket_id='recordings'` to the room's commentator + admin (join on the `{roomId}/` path prefix). Defense-in-depth behind the signed-URL path.

**M-D. ✅ FIXED — No HTTP security headers** — `next.config.ts`
> Fixed: `headers()` now applies `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, and HSTS to all routes. Verified present in-browser. Full CSP + Permissions-Policy deferred to Phase 10 (CSP needs a nonce for the inline theme script; Permissions-Policy intentionally omitted so the LiveKit mic keeps working). Re-audit: closed, no regression. Original detail:

No CSP, `X-Frame-Options`, HSTS, or `X-Content-Type-Options` anywhere → clickjackable, no transport-security header, no script/connect constraint.
- **Fix:** add a `headers()` function (next.config.ts) with `X-Frame-Options: DENY` (or CSP `frame-ancestors 'none'`), `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, HSTS now; defer a full CSP (inline theme script + Supabase/Ably/LiveKit/Sportmonks origins) to Phase 10.
